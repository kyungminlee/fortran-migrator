### 1.4 Notable design observations

- **The classifier is the load-bearing component.** Its
  position-agnostic family discovery is what made `9938731` a one-line
  fix. Documented behavior in `prefix_classifier.py:215-280`. Future
  recipe-level "rename pattern" features (debated in earlier session)
  are unnecessary as long as the symbol universe is correctly seeded
  via `depends:`.
- **`pipeline.py:_apply_extra_renames`** (lines 23-39) is the single
  post-classification rename hook. Three call sites (lines 868, 981,
  1112, 1298) — a sibling `pattern_renames` mechanism would mirror
  this shape, but is not currently planned.
- **C migrator's `_make_rename_substituter`** (lines 682-749) carries
  case-preservation logic for multi-char prefix expansions (e.g. the
  abandoned DD/ZZ era); now mostly idle since prefixes are
  single-character. Worth keeping for any future multi-char target.

### 4.3 CI (`.github/workflows/release.yml`)

**Trigger:** git tag `v*` or manual `workflow_dispatch`.

**Jobs:** `stage` × 3 targets → `build` × 5 compilers × 3 MPI = 45
combinations (minus exclusions) → `release` (creates GitHub Release
with attached tarballs, only on tag push).

**Compilers covered:** gfortran-12, gfortran-15, ifx-2024, ifx-2025,
flang-20.

**Exclusions:** kind10 + ifx (no 80-bit on x86-64 Linux); kind16 +
flang (no KIND=16 yet).

**Critical gap:** the staged artifact does not include `tests/`.
**ctest is never run in CI.** The pipeline produces release tarballs
of compiled libraries but performs no verification beyond linker
success.

### 4.4 Build risks

- Global `-w` (suppress all warnings) — masks potential migration
  bugs. Mitigated by tests, but the tests don't run in CI.
- FetchContent multifloats patches `multifloats/CMakeLists.txt` on
  the fly to handle `CMAKE_SOURCE_DIR` vs `PROJECT_SOURCE_DIR`
  mismatch. Fragile if upstream restructures.
- One observed transient build error during this session
  (`/tmp/stg-mf-fix` ld LTO plugin warning + `Error 2` on first
  multifloats build, succeeded on retry). Not reproducible; likely
  parallel-make race.

## 8. Phase-22 DMD precision divergence — focused dive

The single longest-running test failure on this branch was
`test_zgedmd` and `test_zgedmdq` on kind16 and multifloats (all
layer-(a) sorted-spectrum comparisons), failing by `~4.6e-17`
(precisely fp64 ε). Earlier in this session I attempted to track
down the leak and concluded:

- The migrated `xgedmd.f90` source is clean (only `Z→X`/`D→Q` /
  `WP=real64`→`WP=16` from `zgedmd.f90`).
- Compiling `xgedmd` with `-freal-8-real-16` does not change results
  — confirms there is no missed fp64 step inside the migrator's
  output.
- Promoting `xgeev`, all of `qblas`, `qxblas`, etc. with the same
  flag is also a no-op for the diff.

**Resolution (separate session, commits `36f6f24` / `cfa25fb`):** the
fp64 step is in the **reference build**, not the target. The
upstream Netlib `zgedmd.f90` line 1001 contains:

```fortran
W(i,j) = CMPLX(RWORK(N+i),ZERO,KIND=WP)*W(i,j)
```

With `WP = real64` and gfortran 13 + `-freal-8-real-16`, `KIND=WP`
expands to `KIND=8`. The compiler then truncates the multiplication
to fp64 even though the surrounding storage is aliased to 16 bytes.
Bleeding ~1 fp64 ULP into the Rayleigh quotient causes the eigenvalue
divergence. The migrated xgedmd has explicit `KIND=16` everywhere
and is correct.

**Fix:** `tests/lapack/reflapack/overrides/zgedmd.f90` replaces the
`CMPLX(...,KIND=WP)*W(i,j)` with `RWORK(N+i)*W(i,j)` (mathematically
identical, no intrinsic). The reference build now matches the
migrated build at quad precision.

**Status:** RESOLVED on `tests` branch (`cfa25fb` marks P22
RESOLVED in `tests/lapack/TODO.md`). The `/tmp/stg-q-fix` build
captured before this commit still shows the 2 failures; re-stage
will clear them.

**Generalization:** any future "the migrator's source is clean but
the test fails at fp64 precision" report should suspect the same
class of issue and look at the reference path, not the target.

## Addendum 2 (2026-04-30, post-investigation)

out to be **test-driver bugs**, not migration bugs (commit `4b2d5ee`):

- `blacs_test_blacs_pinfo`: the test treated `BLACS_GET WHAT=10` as
  "broadcast topology" but it's actually `SGET_BLACSCONTXT` (returns
  the context's MPI Fortran handle). BLACS has no GET-able topology;
  it's a one-character argument or set via `BLACS_SET WHAT=0`. Subtest

## Addendum 3 (2026-04-30, 100% pass)

The two remaining failures from Addendum 2 (`pblas_test_pdatrmv` and
`pblas_test_pzatrmv` at uplo=L,trans=T,n=80) were traced to a single
**upstream ScaLAPACK PBLAS bug** in `pdatrmv_.c` line 573 and
`pzatrmv_.c` line 577: the off-diagonal `dagemv_`/`zagemv_` call in
the (uplo='L', trans='T'/'C') branch only passes `one` instead of the
user-supplied `ALPHA`. Every other (uplo,trans) combination uses
ALPHA correctly. Result:

    y_got = y_ref + (1 - alpha) * |L_below^T * x_below|

silently incorrect when `alpha != 1.0`. The test catches it at n=80
(10 block-rows of mb=8) with `alpha = 0.6`; n=40 masks because the
diagonal-block contribution dominates when the block count is
small.

**Fix (commit `50adda9`):**

- Patched copies in `recipes/pblas/source_overrides/{pdatrmv_,pzatrmv_}.c`
  with the single-token swap `one → ((char *) ALPHA)`.
- New `source_overrides:` block in `recipes/pblas.yaml` to wire them in.
- Extended the C migrator (`c_migrator.py` + `pipeline.py`) to honor
  `source_overrides` like the Fortran path does. Previously C-language
  recipes silently ignored `source_overrides` (the field documentation
  in `config.py:165` was Fortran-flavored). Now both languages use the
  same recipe field — closes a doc/code gap.
