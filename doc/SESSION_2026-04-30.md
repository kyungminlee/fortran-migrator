## Categorical fixes (all on kind16, mostly carry to other targets)

### 1. impi preset gained both shortpath bypasses

`cmake/CMakePresets.json`:

```json
"MPIEXEC_PREFLAGS": "-genv;I_MPI_ROOT=/opt/intel/oneapi/mpi/latest;-genv;I_MPI_ADJUST_REDUCE=1;-genv;I_MPI_ADJUST_ALLREDUCE=1"
```

Both env vars force binomial / recursive-doubling paths and skip
Intel MPI's intranode optimised `Reduce_local` / `Allreduce`
shortpaths that crash on user-registered MPI_Op + `REAL(KIND=16)`
payloads. `_REDUCE` was already wired; `_ALLREDUCE` was added this
session and resolved the 8 `*rfs` / `*svx` failures plus a handful
of others that go through BLACS `*GAMX2D / IGAMX2D`.

### 2. Two upstream ScaLAPACK 2.2.3 bugs documented + patched

Both written up in `doc/UPSTREAM_BUGS.md` with full root-cause
analysis, fix sketch, "why upstream tests miss it", and the
in-tree workaround pointer. Patches live in
`recipes/scalapack/source_overrides/`:

* **`p?lanhs.f` IAROW double-advance** — pattern
  `IAROW = INXTROW; IAROW = MOD(IAROW+1, NPROW)` should be
  `IAROW = INXTROW; INXTROW = MOD(INXTROW+1, NPROW)`. 8 sites per
  file, sibling to the existing NPROW=1 underestimate fix.
* **`p?geequ.f` column-scale reduction wrong axis** — three
  `*GAMX2D / *GAMN2D / IGAMX2D` calls after the C(JJA) accumulation
  used `'Columnwise'` (a no-op axis after the prior reduction)
  where `'Rowwise'` is needed to merge across grid columns. Each
  grid column was computing COLCND from its own local block.

### 3. Test-harness bugs

* **PDGBTRF / PZGBTRF IPIV under-allocation** — divide-and-conquer
  merge writes past `DESCA(NB)`; tests bumped to
  `nb + 2*(bwl+bwu)`.
* **PDDBTRS / PZDBTRS workspace under-allocation** — upstream
  `WORK_SIZE_MIN` is too small for the local-update QLAMOV.
* **test_pdpttrf** compared `d_loc, e_loc` directly against LAPACK
  DPTTRF — but PDPTTRF stores the cyclic-reduction representation,
  not LDL^T factors. Rewrote to factor+solve and compare solve
  result, mirroring the already-correct test_pzpttrf.
* **target_p[dz]ggrqf wrapper** sized TAUB on the row axis where
  PxGGRQF's TAUB is column-distributed (`LOCc(MIN(P,N))`). Same
  fix in the matching test driver pre-allocations.
* **test_blacs_pinfo** — assumed `BLACS_GET WHAT=10` returned a
  small char code; it's actually `SGET_BLACSCONTXT` (Fortran handle
  for the comm). Fixed assertion to idempotency only.
* **test_qtrsd2d / test_xtrsd2d** — used M=6 N=5 with the LAPACK
  upper-triangular convention; BLACS `BI_GetMpiTrType` for M>N
  follows the upper-trapezoidal convention. Fixed by switching
  to M=N=5.
* **pblas_grid `grid_exit`** now MPI-barriers before
  `blacs_gridexit` so a fast rank doesn't tear the BLACS context
  down while peers are still inside the user code.

### 1. kind10 / multifloats PBLAS Level 1/2 heap corruption

Tracked in `tests/pblas/TODO.md` (and cross-referenced from
`tests/blacs/TODO.md`).

Failure inventory (kind10):

```
heap-corruption at exit (free invalid pointer):
  pblas_test_pdcopy   pblas_test_pdscal   pblas_test_pdswap
  pblas_test_pzcopy   pblas_test_pzswap

sporadic mid-loop Cblacs_gridinfo returns -1:
  pblas_test_pdagemv  pblas_test_pdasymv  pblas_test_pdatrmv
  pblas_test_pdsymv   pblas_test_pdtrmv   pblas_test_pdtrsv
  pblas_test_pzagemv  pblas_test_pzahemv  pblas_test_pzatrmv
  pblas_test_pzgemv   pblas_test_pzhemv   pblas_test_pztrmv
                      pblas_test_pztrsv
```

multifloats has the same set plus `pdaxpy / pzaxpy / p?scal /
pzdscal` plus a small ScaLAPACK pz tridiagonal cluster (`pzdtsv /
pzdttrf / pzdttrs`).

Key observations from the dig:

* The same test sources pass cleanly on kind16/impi, so the bug is
  in the migrated `e/y/m/w`-prefix BLACS or PBLAS C globals — not
  in the test wrappers.
* Mode 2 is timing-dependent: a binary that fails in ctest passes
  5/5 runs in tight isolation. Adding `fprintf` in `peaxpy_` made
  the bug vanish; removing it kept it gone for that build.
  Suggests memory-layout / heap-state sensitivity rather than a
  deterministic logic bug.
* `Cblacs_gridinfo(0)` returns `nprow=-1` because
  `BI_MyContxts[0]` is NULL on the failing rank. Tracing
  `blacs_gridexit_` showed it is **not** called between
  iterations, so something else is writing NULL into that slot —
  heap corruption.
* multifloats has a parallel blocker: `mana_aux.F:4321` fails to
  build (`'a' argument of 'dble' must have a numeric type`) —
  multifloats fixed-form continuation regression in the migrator.
  Currently bypassed by excluding `mmumps_test_` from ctest.

Suggested next steps:

1. Run valgrind on a single failing kind10 PBLAS L1 binary in
   isolation — ideally with `--track-origins=yes` and the kind10
   workaround for the long-double precision-loss issue (e.g.
   `--vex-iropt-precise-memory-exns=yes`). The relevant trace
   should pinpoint which write corrupts `BI_MyContxts[0]`.
2. Diff the migrated `epblas / mpblas / wpblas` C sources against
   the kind16 `qpblas` to look for a global-state pattern (likely
   one of the `BI_*` extern globals) that the migrator's prefix
   rename collides on.
3. Investigate the `mana_aux.F` fixed-form continuation regression
   for multifloats — `tests/mumps/TODO.md` already has an entry on
   it; that has to be fixed before mf MUMPS tests can run, which
   in turn unblocks the full mf ctest.

## Dead ends (don't repeat)

* Adding `mpi_barrier` at the **start** of a single PBLAS wrapper
  fixes that wrapper but doesn't generalise — pdgemv started
  passing one moment, then passed again later without the barrier
  on the next rebuild. Heisenbug.
* `MALLOC_CHECK_=0` does not paper over the `free(): invalid
  pointer` mode 1 crash; the corruption is real.
* Adding the barrier in `pblas_grid::grid_exit` does **not** fix
  the mid-loop context-loss — by then the corruption has already
  happened. Kept the barrier anyway because it's clearly the right
  thing for the PBLAS-doesn't-synchronise pattern; just doesn't
  fix this specific bug.
